# Agent IAM（智能体身份与访问管理）

> 在 Agent 系统里，真正的安全边界不再只是网络或账号本身，而是“谁发起、Agent 代表谁、凭证藏在哪里、当前操作为什么合理”的整套身份与授权治理。

## 核心要点

- **身份成为主边界**：当 Agent 同时能读私有数据、访问外网、处理不可信内容时，传统外围边界很容易失效，身份与授权成为最后一道硬边界。
- **传统 IAM 不够用**：静态角色、宽泛 OAuth Scope、长期密钥和单一服务账号，和 Agent 的动态任务链、委托链、多跳执行天然冲突。
- **核心不只是认证**：成熟的 Agent IAM 至少要同时处理身份传播、无秘钥验证、上下文感知和意图感知授权。
- **本质是把“代理关系”显式化**：系统需要知道“哪个 Agent 在替哪位用户执行什么动作”，否则审计、归因和最小权限都会失真。

## 详细说明

### 为什么 Agent 时代会把 IAM 推到前台

安全研究里常用“致命三要素”来描述高风险 Agent：

- 能访问私有数据
- 能向外通信
- 会接触不可信内容

一旦三者同时成立，Agent 又是概率式行为体而不是确定性程序，风险就不再只是“某个接口有没有鉴权”，而是“整条委托链有没有被正确约束”。

### 传统 IAM 为什么在 Agent 面前失灵

- **身份传播断裂**：请求经过多层 Agent / subagent 之后，资源侧往往只看到最后一跳，无法还原最初发起者，容易产生“混淆代理人”（Confused Deputy）问题。
- **静态权限不适配动态任务**：人类角色通常稳定，而 Agent 的权限需求会随任务上下文快速变化。给它长期高权限，攻击面会迅速膨胀。
- **Scope 颗粒度过粗**：传统“读写邮件”“访问文档库”对 Agent 来说太宽，缺少“只能读某来源、只能写某目标”的任务级约束。
- **密钥暴露面过大**：当 Agent 同时具备文件读写、命令执行和联网能力，把长期密钥直接交给它，基本等于默认接受泄露风险。

### Agent IAM 的四个支柱

#### 1. 身份传播（Identity Propagation）

让用户身份上下文穿透 Agent 链路，一直传到真正访问资源的那一层。Agent 不该只拿一个通用高权服务账号到处跑，而应代表具体发起用户行事。

#### 2. 无秘钥验证（Secretless Auth）

把“密钥的持有”和“密钥的使用”拆开。密钥放在 Agent 无法直接读取的外部安全环境里，Agent 只拿引用或短期票据，请求在受控网关处动态注入凭证。

#### 3. 上下文感知（Context Awareness）

授权不只看“你是谁”，还看“你现在处在什么执行环境、会话状态和行为时序里”。这让系统能基于运行时状态拒绝跳步执行、异常时段访问或异常环境调用。

#### 4. 意图感知授权（Intent-Aware Authorization）

系统不仅判断 Agent “能不能做”，还要判断它“为什么做”。如果当前操作和用户原始目标不一致，哪怕权限名义上足够，也应该被拦截。

## 上下文

这套框架在 2026 年的 Agent 安全讨论里迅速升温。相关文章用 OpenClaw 的高权限、暴露实例与网页触发执行风险作为案例，说明自托管 Agent 一旦进入真实业务环境，安全问题就会从“工具权限”升级成“身份委托治理”问题。
