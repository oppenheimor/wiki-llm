# Tool Execution Pipeline（工具执行管线）

> 从“模型表达工具调用意图”到“工具真正执行并把结果返回给模型”之间的分层处理概念，通过验证、校准、拦截、授权和结果整形，把不可信的模型输出转化为可控的系统动作。

## 核心要点
- **模型生成的工具输入默认不可信**：即便 JSON 格式合法，也可能类型错、路径假、语义不成立，甚至请求高风险操作。
- **单次工具调用不是一步，而是一条联锁管线**：schema 验证、业务校验、输入补全、前后置 Hook、权限判定、执行与结果处理各自承担不同职责。
- **错误信息的受众变成了模型**：好的错误不是底层错误码，而是能帮助模型立刻自我纠正的可操作反馈。
- **上下文预算是工具系统的一部分**：结果截断、落盘引用、总输出预算直接决定长会话中的稳定性、延迟和缓存命中率。
- **同一概念可以有不同工程形态**：Claude Code 倾向顺序流水线，OpenClaw 可用事件驱动拆职责，OpenCode 则展示了应用层 batch 并发的变体。

## 详细说明

### 1. 先验证格式，再验证语义

第一层通常是 schema 校验，确认 JSON 结构、字段类型和必填项合法；Zod 这类运行时 schema 系统常被用于这一层。第二层则做业务校验，确认参数在工具语义上真的可执行，例如路径存在、编辑锚点唯一、参数组合满足前提。

这两层的目标不是“挡错”本身，而是把失败转成可修复反馈。相比 `validation failed` 或 `ENOENT`，更高价值的返回是“`file_path` 应为 string 却收到 number”或“文件不存在，但当前目录下这些候选路径可能是你要找的”。

### 2. 输入补全与标准化

通过验证后，系统常会把相对路径转成绝对路径、补默认值、派生运行态字段。一个重要的工程细节是：很多系统不会直接篡改模型原始输入，而是保留原始序列化内容用于对话历史与缓存，同时生成一份补全后的副本用于执行阶段。

### 3. 前后置 Hook

PreToolUse Hook 在执行前运行，可用于改写路径、阻止命令、附加策略或审计信息；PostToolUse Hook 在执行后运行，可用于过滤敏感输出、自动运行 lint/format、记录日志或触发后续流程。Hook 让“项目级约束”不必硬编码进核心框架。

### 4. 权限检查

成熟系统往往把权限做成多层决策：先做静态规则匹配，再对未覆盖场景用轻量分类器判断风险，最后才在必要时向用户交互式询问。这样能把只读和低风险操作自动化，把真正危险的动作留给人工授权。

### 5. 真正执行之后，还有结果预算与错误整形

工具结果不是越完整越好；如果直接把海量输出灌回上下文，会压垮窗口预算。于是系统会给单工具结果和单轮总结果设预算，超限时把完整内容落盘，只把摘要和读取路径返回给模型。

错误处理同理。Agent 系统里的错误设计不再只是“方便开发者排查”，而是“方便模型继续下一步”；这决定模型会不会在同一个问题上反复猜测。

### 6. 流水线、事件驱动与 Batch 变体

Claude Code 这类系统更容易被理解为线性流水线：验证 → 校验 → 标准化 → Hook → 权限 → 执行 → 后处理。OpenClaw 也可把同样职责拆成事件处理器，让多个监听器围绕工具执行生命周期协同。OpenCode 的 Batch Tool 则进一步说明，并发不一定依赖模型原生 multi-tool 输出，也可以由应用层把多个子调用封装成一个元工具来并发调度。

## 上下文

这个概念是理解 Coding Agent 工程化差异的关键抓手。它解释的不是“模型怎么调用工具”，而是“系统怎样把模型的工具调用意图安全、稳定、低成本地变成现实动作”。
