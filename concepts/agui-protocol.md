# AGUI Protocol

> Agent 与图形界面之间用于传输流式文本、工具调用状态和结构化交互事件的协议层，让前端不再只能“按字符显示文本”，而能理解“当前正在发生什么”。

## 核心要点
- **它解决的是 Agent 输出的“语义缺失”问题**：纯文本 SSE 只能看到字节流，看不到这段内容是普通回复、工具参数、工具结果还是阶段切换事件。
- **核心不是“能流式”，而是“能分类型地流式”**：文本开始、文本增量、工具输入开始、工具输入完成、工具输出可用，这些都需要单独事件类型。
- **前后端的边界因此更清晰**：后端负责把 Agent 执行轨迹编码成带类型的事件流；前端负责按事件类型映射为文本块、Tool Card、审批组件或状态提示。
- **它是 Agent UI 的协议层，不是 Agent 推理层**：不会改变模型如何思考或调用工具，但会直接决定用户是否看得懂 Agent 在干什么。
- **一旦协议稳定，渲染层就能框架无关复用**：React、Vue 等前端只要会消费同一种事件流，就能共享同一套 Agent 交互语义。

## 详细说明
### 为什么“纯文本流”不够
很多 Agent 后端一开始只会返回逐段文本，这对聊天机器人够用，但对 Agent 不够。因为 Agent 执行过程中还包含工具参数生成、工具调用中、工具结果返回、阶段切换等状态，如果这些都被压平为一串文本，前端只能“猜”。

### 协议真正提供的能力
AGUI 这类协议把输出流改成结构化事件流。典型事件包括：
- 文本块开始 / 增量 / 结束
- tool input 开始 / 增量 / 可用
- tool output 可用

这意味着前端拿到的不是“字符”，而是“可解释的交互事件”。

### 它与工具调用的关系
函数调用或 tool calling 解决的是“模型如何向系统发起动作”；AGUI 解决的是“系统怎样把这个动作过程可视化给用户”。两者常常同时出现，但不是一回事。前者偏模型接口，后者偏用户界面协议。

### 协议层为什么会影响产品体验
同样一个 Agent loop，如果前端只能展示一坨文本，用户会觉得系统很黑盒；如果能按工具、状态、结果分块展示，用户就更容易理解当前阶段、判断是否需要干预，并建立对系统行为的信任。

## 上下文
该概念通常出现在“把 Agent 从命令行/纯文本聊天升级为可视化产品”的阶段。它处在 [[concepts/agent-loop]] 与前端渲染层之间，是 Agent 产品化里非常容易被低估但体验收益极高的一层。
