# Token Efficiency

> 在大模型与 Agent 系统里，用更少 token 完成同等质量任务的能力，不只是成本优化，而是直接决定系统能否规模化和具备竞争力的核心约束。

## 核心要点
- **它是算力预算问题，不只是账单问题**：token 消耗最终会转化为 GPU 占用、延迟、并发能力和服务成本。
- **长 session 的浪费会指数放大**：多轮低价值工具调用、巨型上下文回传、糟糕的 resume 与前缀设计，会把浪费从单次请求扩散到整段会话。
- **它依赖架构协同**：单靠模型或单靠推理引擎都不够，Agent 框架、推理引擎和缓存策略必须协同设计。
- **它是进攻性指标**：谁能在相同质量下把 token 消耗降一个数量级，谁就能在同样 compute budget 下服务更多用户。

## 详细说明
在普通软件里，资源浪费常常会被硬件进步不断吞掉，比如内存膨胀在很多场景里最终只是“电脑更卡一点”。但在 LLM 系统里，token 浪费的代价更直接，也更难被供给侧快速对冲。因为每个额外 token 都对应注意力计算、KV Cache 占用、GPU 时间片和真实金钱成本。

这也是为什么很多 Agent 产品看上去“功能已经能跑”，但一旦放进真实长 session、高并发或本地推理场景，就会暴露出惊人的低效。问题往往不在单次调用，而在整个会话构造方式上，例如：
- 明明可以复用前缀，却每轮都重发大段上下文
- 明明可以压缩历史，却保留大量低价值中间痕迹
- 明明只需少量工具交互，却因为编排粗糙触发一串昂贵调用

因此，token efficiency 更准确的理解不是“省钱技巧”，而是系统设计是否真正尊重算力边界的试金石。
