# Model-Native Agent Interface（模型友好接口模式）

> 不要把系统内部的复杂数据结构原样暴露给模型，而要把能力翻译成模型天然熟悉的表示与交互面，再随着工具规模增长做渐进式披露。

## 问题场景

很多团队第一次做 Agent 时，会直接把内部 API、对象模型、查询 DSL 或工具注册表完整暴露给模型。结果往往是：

- 模型能调用，但很不稳定
- prompt 需要塞进大量解释性上下文
- 工具一多，选择质量和延迟一起恶化
- 团队只能围着一个巨型 prompt 字符串小心修补

## 解决方案

### 1. 优先给模型它“本来就会”的表示

- 用普通 Markdown 承载文档结构，而不是强行暴露内部块级 XML
- 用 SQL 查询表，而不是把复杂 JSON 查询 DSL 直接丢给模型
- 用简洁、稳定、语义明确的工具定义，而不是让模型理解整套内部实现细节

核心原则不是“忠实映射内部系统”，而是“让模型更容易稳定完成任务”。

### 2. 把接口设计成可被 Agent 自举和自修复

某些场景里，CLI 比远程协议更适合 Agent，因为它和执行环境共处一地，出错时 Agent 能自己调试、自己修补、自己扩展缺失能力。这是一种“能力表面”和“修复表面”对齐的设计。

### 3. 工具多了以后做渐进式披露

当工具数量扩大到几十上百个时，不要一次性全塞进上下文。应让 Agent 先搜索、过滤、再加载候选工具，否则 token 成本、决策质量和误用概率都会恶化。

### 4. 用 eval 反向塑造接口

接口是不是“模型友好”，不能靠直觉宣布，必须通过 eval 看稳定性、通过率、延迟、P99 失败轨迹以及供应商差异。好的接口设计，往往是和 eval 体系一起长出来的。

## 权衡取舍

- 对内部模型做“翻译层”通常意味着一定的信息损失，不再追求无损映射
- CLI 往往比 MCP 拥有更宽的执行面，因此默认权限边界更难收紧
- 渐进式披露需要额外的索引、搜索和工具编目基础设施
- 接口变简单后，系统复杂度并没有消失，只是被移动到了更可控的位置
