# Tool-Aware Streaming UI

> 让 Agent 前端按“文本块、工具调用、工具结果、状态事件”等不同语义单元分别渲染，而不是把整个执行过程压平成一条文本流。

## 问题场景
当 Agent 只返回逐字流式文本时，用户通常只能看到“模型在输出字”，却看不到：
- 是否正在调用工具
- 工具参数是否还在生成
- 工具是否已经执行完成
- 当前显示的是普通回复，还是某个工具的输出

这会让 Agent 在功能上能跑，但在体验上仍像“会联网的聊天框”。

## 解决方案
### 1. 先把后端输出升级为结构化事件流
不要把所有执行轨迹都塞成纯文本。后端应输出带类型的流式事件，让前端知道这是 text part、tool input 还是 tool output。没有这层协议，前端就无法稳定渲染不同组件。

### 2. 前端以 part 为中心渲染，而不是以 message string 为中心
消息不应再被视为一整段 markdown 字符串，而应被拆成多个 part。文本 part 走 markdown 组件，tool part 走专属工具卡片，pending / error / success 状态分别映射不同视觉组件。

### 3. 文本和工具渲染要分层处理
文本流可以继续做 markdown、代码块、表格、Mermaid 等富渲染；工具流则关注状态、输入、输出和结果摘要。两类渲染诉求完全不同，混在同一个组件里通常会越写越脏。

### 4. 新增工具时只扩展映射层
理想形态不是“每加一个工具都改整套流逻辑”，而是保留统一协议，只在前端增加 `toolName -> renderer` 的映射。这让系统更接近插件化。

## 权衡取舍
- 协议与渲染层会比纯文本聊天更复杂，需要维护事件类型和组件映射
- 不同工具输出结构不一致，前端需要定义每类工具的可视化约定
- 如果工具种类迅速膨胀，没有统一的设计规范，UI 会从“丰富”滑向“杂乱”
- 协议一旦对外暴露，后续演进需要考虑兼容性
