# Benchmark First Agent Development

> 在 Agent 或 AI 系统开发中，先搭可复现、可比较、能拦回归的评估框架，再谈功能和优化，用度量来替代“看起来变好了”的自我感动。

## 问题场景
Agent 系统极易陷入 demo 驱动的错觉：功能越来越多、提示词越来越复杂、策略越来越花，但没人能回答一个最基本的问题，这些改动到底有没有让结果更好。

尤其在 Agent 领域，评估往往比实现更难。实现一个新策略可能很快，但如果没有稳定 benchmark，就很难区分：
- 真正提升
- 样本挑选带来的幻觉
- 对某些 case 有利但整体无效的局部优化
- 牺牲准确率换来的表面性能提升

## 解决方案
把 benchmark 放在项目极早期，甚至第二天就开始建设。核心动作包括：
- 先定义成功指标，再开始堆功能
- 给准确率与性能都建立 CI，而不是只看单一数值
- 让阈值基于真实 run-to-run variance，而不是凭感觉
- 在功能迭代中持续用同一套框架回归，避免“每次都换标尺”

这个模式的价值，不只是帮团队判断“有没有变好”，更重要的是让团队停止围绕感觉做决策。

## 权衡取舍
先做 benchmark 会延缓短期 demo 速度，也会让很多“看上去很 promising”的功能在进入主线前就被否掉。但这恰恰是它的价值所在：把资源留给能被证明确实有效的方向。
