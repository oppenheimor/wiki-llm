# Same-Content Style Wall Screening

> 在 UI 风格决策前，先把同一份产品内容骨架渲染成多种静态视觉样品并单页横向对比，用可比样品先筛方向，再进入真实页面实现。

## 问题场景
当一个项目手头有大量可选 UI 风格时，团队很容易陷入两种低效状态：
- 直接凭抽象描述选风格，但每个人脑中的“glassmorphism / corporate / brutalist”都不是同一个东西
- 过早挑一个方向深入实现，结果发现问题其实不在功能，而在最上游的视觉路线判断

这类问题在 AI 驱动的前端生成场景里更明显，因为“先做几个版本看看”变得非常便宜，真正稀缺的反而是比较框架本身是否公平、稳定、可复用。

## 解决方案
先定义一份**固定内容骨架**，明确哪些信息在所有样品里保持不变，例如：
- 产品主题
- 标题与摘要
- 核心价值点
- CTA 文案
- 目标人群说明

然后只允许**视觉变量**发生变化，例如：
- 字体气质
- 配色
- 边框 / 圆角 / 阴影
- 纹理 / 背景 / 装饰图形
- 卡片布局节奏

最后把这些样品集中到**同一页面**里横向展示，而不是分散成多个独立页面。这样做的关键价值不是“做了很多 demo”，而是让评审时看到的差异尽量只来自风格本身，而不是内容、结构和交互复杂度混杂后的综合噪声。

在 `enterprise-ai-survey` 项目的这次实践里，这个模式被落成了一个 15 卡片的静态样品墙：主题统一为“企业 AI 转型问卷”，每张卡片都使用同一套信息结构，仅切换风格 preset，以便快速筛掉不合适的视觉方向。

## 权衡取舍
这个模式会主动牺牲一部分“单个方案的完整度”：每个风格只展示一个压缩后的静态切面，不代表其在完整产品中的全部表现。

但它换来的好处非常直接：
- 更早暴露风格方向错误，而不是在完整实现后返工
- 让设计讨论聚焦于视觉语言，而不是被文案、流程、数据状态带偏
- 为后续真实页面实现提供更清晰的候选池

它的限制也要明确：
- 如果内容骨架本身设计得不公平，比较结果会失真
- 某些风格在卡片尺度上表现很好，落到长页面或复杂流程里未必仍然成立
- 这更像“方向筛选器”，不是最终验收器
