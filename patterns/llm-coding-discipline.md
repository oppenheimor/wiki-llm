# LLM Coding Discipline（LLM 编码四纪律）

> 一套针对 LLM 编程代理的行为约束 —— 用「谨慎换速度」的偏置来抵消大模型的常见失误：自作主张、过度复杂化、误伤无关代码、目标漂移。

来源：forrestchang 的 [andrej-karpathy-skills](https://github.com/forrestchang/andrej-karpathy-skills) 仓库根目录 `CLAUDE.md`，作为项目级 Claude Code 行为指令使用。

## 设计意图

明确写在文档开头的 tradeoff：**这套规则有意偏向「谨慎」而非「速度」**。琐碎任务可以绕过；但凡涉及不确定性、改动现存代码、模糊目标的场景，按规则走。

底层洞察：大模型在编码任务上的高频失败模式不是「写不出代码」，而是 ——
1. 默默选了一个解释，没说出歧义
2. 写得太多、太抽象、太"灵活"
3. 顺手"改进"了无关的相邻代码，留下意外副作用
4. 把"让它工作"当成成功标准，没有可验证的终点

四条纪律一一对应这四种失败。

## 四条纪律

### ① Think Before Coding（先思考后写代码）

**不要假设。不要藏起困惑。把权衡摊到台面上。**

实施前：
- 把假设显式说出来；不确定就问
- 存在多种解读时，全部呈现 —— 不要默默挑一个
- 如果有更简单的方案，说出来；该顶回去时就顶回去
- 哪里不清楚，停下，命名困惑点，发问

### ② Simplicity First（先求简）

**用解决问题的最小代码量。不要任何投机性的东西。**

- 不写超出要求的功能
- 不为一次性代码做抽象
- 不加未被请求的"灵活性"和"可配置性"
- 不为不可能发生的场景写错误处理
- 200 行能压成 50 行就重写

自检：「资深工程师会说这写得过度复杂吗？」如果会，简化。

### ③ Surgical Changes（外科手术式改动）

**只动必须动的。只清理自己留下的烂摊子。**

编辑现存代码时：
- 不"改进"相邻代码、注释、格式
- 不重构没坏的东西
- 匹配现有风格 —— 哪怕你会用别的写法
- 注意到无关的死代码 —— 提一句，**不要删**

当你的改动产生孤儿（imports/variables/functions）：
- 移除被你的改动弄成未使用的
- 不要顺手清理预先存在的死代码 —— 除非被要求

测试标准：**每一行变更都能直接追溯到用户的请求**。

### ④ Goal-Driven Execution（目标驱动执行）

**定义成功标准。循环直到验证通过。**

把任务转化为可验证目标：
- "Add validation" → "为非法输入写测试，让它通过"
- "Fix the bug" → "写一个能重现 bug 的测试，让它通过"
- "Refactor X" → "确保改动前后测试都通过"

多步任务陈述简短计划：

```
1. [步骤] → 验证：[检查]
2. [步骤] → 验证：[检查]
3. [步骤] → 验证：[检查]
```

强成功标准让你能独立循环；弱标准（"让它工作"）会让你不停回头要澄清。

## 生效信号

文档结尾给出了"这套规则起作用"的可观测指标：
- diff 里的非必要改动减少
- 因过度复杂而被推翻重写的次数减少
- 澄清问题出现在实施**之前**，而不是出现错误**之后**

## 与其他 Claude Code 配置层的关系

按 [[patterns/claude-code-config-layers]] 的四层模型，本套纪律属于 [[concepts/claude-code-rules]]（规则层）—— 即写入 CLAUDE.md 由 LLM 自行遵守的指令式约束，而非由 harness 强制执行的 hooks。其特点：

- 不是工具调用约束，是行为风格约束
- 不依赖任何外部脚本，纯 prompt 注入
- 适合作为团队/个人通用 base，再叠加项目特定 CLAUDE.md
