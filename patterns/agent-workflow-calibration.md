# Agent Workflow Calibration（Agent 流程校准）

> 好的 Agent 工作流不是把所有好习惯串成一条僵硬命令链，而是把约束放在真正高风险的节点上，让流程既能控质量，又能持续执行。

## 核心判断

很多团队给 Agent 写流程时，容易犯三类错误：
- 把不同层级的东西写在一起：协作模型、Git 策略、需求管理、验证方式、归档动作混成一条线性口号
- 把“推荐默认”写成“无条件强制”：所有任务都必须完整走同一套重流程
- 把“严谨”误写成“全都清零”：所有 review 评论、所有测试类型、所有检查都被升级成统一阻塞项

流程一旦这样写，问题不是“不够规范”，而是**不可执行**。团队会在真实任务里绕开它，最后留下的不是纪律，而是形式主义。

## 校准原则

### 1. 先区分“什么时候适用”，再定义“怎么执行”

OpenSpec、评审、e2e、并行开发都不是“永远全部开启”的开关。更合理的写法是先定义触发条件：
- 改行为、改流程、改数据模型、改接口、改外部集成：进入规格驱动流程
- 纯文案、小修复、不改预期行为：允许轻流程，但仍要声明验证方式
- 核心用户路径、跨页面流程、高回归风险：补 e2e
- 文件边界、接口边界、交付顺序清楚：允许并行

先定义适用条件，流程才会变成风险控制，而不是口头上的“态度正确”。

### 2. 并行策略要和隔离策略绑定，而不是和仓库拆分手段混写

多智能体并行开发的关键不是“多开几个 Agent”，而是**如何隔离上下文和代码改动面**。如果写成 Agent Teams + git subtree，常常混淆了“协作组织方式”和“仓库内容同步方式”。

更稳定的表述通常是：
- Agent Teams / Subagent：谁协作、谁负责
- git worktree：在哪里并行、怎么隔离工作区
- merge / review / verification：如何回到主线

把这三层分开写，流程会清楚很多。

### 3. 验证要有基线，也要允许按风险加码

“单测 + e2e 全写齐”听起来很严密，但在探索型任务或中小变更里经常不现实。更好的规则是：
- 先定义默认基线：例如 lint、typecheck、build、unit tests、spec validate
- 再定义加码条件：哪些任务需要 e2e、手工 smoke、专项检查

这样流程既不会漏基本盘，也不会把每个任务都抬成重型 QA 项目。

### 4. review 要区分 blocker 和非 blocker

如果把所有 review 评论都要求“清零”，流程很快会退化成无休止拉扯。真正有用的写法是：
- blocker 必须解决
- 非阻塞建议要么当轮处理，要么记录暂缓原因

这样既保留质量门槛，也避免让风格偏好冒充发布闸门。

### 5. archive 必须写在流程尾部，而不是和 propose / apply 并列成前置口号

`propose -> apply -> archive` 作为生命周期没问题，但如果写成“做之前严格按这套流程执行”，就会让时序变得含混。归档的本质是**在实现、验证、规格一致之后收尾**，不是进入任务前的宣誓动作。

## 推荐写法

一段更可执行的 Agent 流程，通常至少要分成四块：
1. 规划与拆分：什么时候可以并行，什么时候先串行收敛
2. 规格管理：哪些任务必须进 OpenSpec，哪些可以轻流程
3. 验证策略：默认基线是什么，什么情况下追加 e2e 或专项检查
4. 完成定义：review、验证、规格一致、归档分别在什么时点发生

本质上，这是把“线性口号”改写成“条件化策略”。

## 适用场景

- 想给 Agent 写项目级工作流，但担心规则过重、最后没人真正执行
- 团队已经有 OpenSpec、review、测试等纪律，但它们在文档里写得互相打架
- 多智能体协作场景下，需要把协作模型、工作区隔离和验证回收拆开表达
