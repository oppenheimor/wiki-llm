# Prompt As System Design

> 在 Agent Coding 场景里，prompt 不只是给模型下命令的文本，而是把模块边界、抽象粒度、扩展方向和约束条件编码成系统设计的一种表达形式。

## 问题场景
很多人会把 AI 生成的大段代码理解成“模型自动设计了系统”。但在复杂重构和长期可维护项目里，真正决定结果质量的往往不是模型会不会拆文件，而是人有没有先想清楚：
- 哪些维度未来会变化
- 哪些抽象值得提早做
- 哪些差异不能被粗暴统一
- 复用与灵活性的边界在哪里

## 解决方案
把 prompt 当成系统设计的承载物，而不是“随手写几句指令”。也就是说，prompt 里应当编码的是：
- 模块拆分原则
- 未来变化预期
- 明确的不变项与可变项
- 抽象边界与禁止过度抽象的约束

在这种模式下，模型给出的不是“凭空设计”，而是把已有设计高速展开成实现细节。AI 负责分解与执行，人负责给出那个正确的分解方向。

## 权衡取舍
这个模式要求操作者先具备足够清晰的系统理解，前期思考成本更高；但它能显著减少“重构很快完成、方向却完全错了”的高代价事故。相反，如果把 prompt 只当成一句“帮我重构一下”，模型通常只能给出统计平均意义上的拆法。
